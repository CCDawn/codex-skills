# BRT Output Forms

仅在用户需要看见对齐、正式方案/handoff、自然闸门或破坏性确认时使用。SILENT/MICRO 不套模板。

## ALIGN

```text
需求对齐:
- 我理解的结果: ...
- 我准备做: ...
- 不包含: ...
- 完成证据: ...
- 最需要纠偏: ...
默认推进: ...
```

只有回答不改变实现的 `HIGH/MEDIUM` 才默认不要求回复；触发 `Alignment Value Gate` 时等待用户回复“按推荐”或纠正选项。

## LOW Collaborative Calibration

先说明当前理解、依据和可能错在哪里，再一次提出 2-4 个相关问题：

```text
1. <高影响问题>
   推荐: ...；理由: ...
   其他选择会导致: ...
   错判影响: ...
```

末尾说明：可回复“按推荐”，或只纠正不对的编号。不得使用“简单版/完整版/更灵活”等无行为差异选项。

问题必须共同决定一份可执行契约；提问前先摘要已查到的项目惯例。能由窄范围本地证据回答、答案不改变实现或只是满足 Agent 好奇心的问题不得询问。

## BLOCKED

```text
当前阻塞: <原因和影响>
已确认: <证据>
恢复路径: <可执行方案>
需要你确认: <一个不可约问题>
```

## 方案或交接

只保留会被下一执行者使用的字段：目标与非目标、owning surface、关键顺序/决策、保护边界、验证、风险、Next Action。没有真实拆分就不输出 Task Graph；没有持久状态就不输出 Ledger。

## 多动作队列

```text
执行顺序:
1. <动作> [SAFE_DIRECT/PLAN_THEN_EXECUTE/DEFERRED/BLOCKED]；证据；完成条件
```

已有执行许可时连续推进 `SAFE_DIRECT`；仅在 BLOCKED、真实取舍或高风险授权处停下。

## 破坏性确认

说明具体对象、不可逆影响、恢复/备份状态，并要求用户输入明确确认词。不要把普通提交、低风险编辑或可逆清理包装成破坏性菜单。

## 完成

```text
已完成: <行为结果>
验证: <命令/检查与结果>
剩余风险: <无或真实缺口>
下一步建议: <一个具体动作>
```

`下一步建议` 是最后一行；只有自然闸门才提供 2-3 个具体选项。

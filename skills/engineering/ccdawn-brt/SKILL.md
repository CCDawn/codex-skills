# BRT

## 核心目标

BRT 的第一目标是对齐用户心里真正想要的结果，不是展示流程。

把用户说出口的入口、感受或局部方案，收敛成：

- 行为：谁在什么条件下看到什么结果；
- 评审：哪些视角会挑战这个行为；
- 测试：怎样证明行为达成；
- 执行：是否可以安全进入实现。

## 使用原则

- 不等待用户把需求说完整；先主动揣测。
- 不把用户原话直接当需求；先翻译成可观察意图。
- 默认给 2-3 个有判断价值的候选意图，并标出推荐项。
- 不凑陪衬选项；没有真实取舍价值时只给 1-2 个。
- 可以一次问多个问题；默认 2-4 个高信号选择题。
- 每个问题都要给推荐答案、选择信号和取舍代价。
- 能从代码、文档、测试、配置或日志查到的，先查再问。
- 低风险、可逆、可本地查证的问题，可以声明假设后继续。
- 改代码前必须完成必要对齐；除非用户明确要求跳过。

## 何时触发

遇到这些内容时主动使用 BRT：

- 模糊需求、宽泛词、局部方案、用户只表达感受；
- 默认行为、用户应该看到什么、边界情况、失败路径；
- 状态流转、生命周期、重试、取消、恢复、回滚；
- 权限、隐私、安全、密钥、数据保留；
- 记忆、恢复、续做、再次打开；
- 验收标准、BDD、Given/When/Then、怎么测试；
- agent 行为、工具选择、计划、交接、自主工作流；
- promotion/apply/rollback、持久化、公共 API 契约；
- 长任务、多步执行、阻塞恢复。

明显拼写修复、纯机械重构、依赖升级、没有行为歧义的小型内部 helper 改动，不要使用 BRT。

## 对齐循环

1. 读上下文：先看相关代码、测试、文档、日志、配置或历史决策。
2. 主动揣测：给出 2-3 个候选意图，说明各自适合什么情况和你推荐哪一个。
3. 设计问题：只问会改变行为、范围、风险、测试或实现许可的问题。
4. 提供选项：每个问题给 2-3 个互斥选项，并标出推荐答案、适用信号和代价。
5. 更新账本：用户选择、纠正或说“按推荐来”后，更新需求账本。
6. 继续推进：足够对齐就进入行为、评审、测试或执行闸门；不够就继续追问或 probe。

## 候选意图

候选意图用于帮用户选择，而不是让用户重新写需求。

默认输出：

- A 保守理解：最小、可逆、风险最低；
- B 标准理解：最可能满足用户真实目标，通常作为推荐；
- C 扩展理解：更完整但范围更大，只有有价值时输出。

候选质量门槛：

- Goal：想解决什么；
- Expected Output：用户最终看到什么；
- Required Capability：需要什么能力或改动。
- Choose When：什么信号说明用户该选它；
- Tradeoff：相对推荐项牺牲什么或增加什么成本。

如果某个候选只能写成“更简单”“更完整”“范围更大”而说不出具体行为差异、适用信号和代价，删除它或合并到推荐项说明里。

## 高信号问题

问题必须影响至少一项：

- 用户最终可观察结果；
- 范围边界；
- 读写、删除、发布、回滚、权限或安全风险；
- 数据保留、状态迁移、默认值或失败路径；
- 测试层级、验收证据或是否允许实现。

问题设计规则：

- 默认 2-4 个；低风险小任务可 0-1 个，高风险任务最多 5 个；
- 按影响排序：目标 > 范围 > 风险/权限 > 输出形态 > 测试验收；
- 每个问题只解决一个决策点；
- 每个问题给 2-3 个互斥选项；
- 每个选项都写清楚适合场景或取舍，不只给名词标签；
- 标出推荐选项和原因；
- 非推荐选项必须说明用户在什么情况下应该选它；
- 允许用户直接说“按推荐来”。

不要问：

- 能从本地上下文查到的问题；
- 只影响实现细节、不影响行为契约的问题；
- 多个等价命名、措辞或展示顺序；
- 过早的完整方案填空。
- 为了凑数量而生成的泛泛选项。

## 输出形态

默认用短格式：

```text
需求对齐:
- 我猜你真正想要的是: ...
- 候选意图:
  - A ...；适合...；代价...
  - B ...（推荐）；适合...；代价...
  - C ...；适合...；代价...
- 关键信号: ...

请选关键项，或直接说“按推荐来”:
1. [问题]: A 适合... / B（推荐）适合... / C 适合...
2. [问题]: A（推荐）... / B ...
```

高风险或即将实现时，补充：

- 范围边界：本轮做什么，不做什么；
- 行为摘要：触发条件、可观察结果、失败路径；
- 自审结论：最相关视角的质疑点、通过证据和结论；
- 测试锚点：最低但足够的验证方式；
- 执行闸门：允许动作、限制、验证命令或证据。

需要完整模板或反例时，读取 `references/templates.md`。

## 需求账本

非单点小改必须维护需求账本，默认只输出摘要。

账本字段：

- 用户原话；
- 候选意图；
- 已确认意图；
- 用户目标；
- 可观察结果；
- 范围边界；
- 操作边界；
- 失败路径；
- 验证证据；
- 显式假设和剩余风险。

用户回答后先更新账本，再继续规划或实现。

## 评审与测试

对齐后再做评审和测试，不要把它们放在意图发现前面。

评审视角从风险推导，默认考虑：

- 核心用户；
- 维护者；
- 测试/QA；
- 必要时补充安全/隐私、运维/支持、集成方。

多角度自审必须产出 Review Matrix，而不是只列角色。

默认选择 2-4 个最相关视角。每个视角只写：

- Challenge：这个视角最可能反对什么；
- Evidence：什么证据说明可以通过；
- Verdict：PASS、NEEDS_CLARIFICATION、NEEDS_CHANGE 或 ACCEPT_RISK。

Evidence 必须是具体证据，至少来自以下之一：

- 用户选择或确认；
- 本地代码、文档、测试、日志、配置或历史决策；
- 可执行验证命令、测试锚点或可逆 probe；
- 明确假设，但必须同时写出风险和后续验证方式。

无效 Evidence：`应该可以`、`看起来没问题`、`风险较低`、`后续再测`、只复述推荐理由、只说 agent 自己会小心。

如果任何视角是 NEEDS_CLARIFICATION 或 NEEDS_CHANGE，不要进入实现；先追问、调整行为契约或缩小范围。ACCEPT_RISK 必须说明被接受的剩余风险。

测试选择最低但足够的层级：

- 单元：规则、解析、策略、状态决策；
- 集成：持久化、文件系统、多模块协作、agent 工作流；
- 端到端：必须通过真实链路才能证明的行为；
- 人工验收：视觉、探索性或暂不适合自动化的行为。

## 执行闸门

改代码前确认：

- 已确认意图；
- 范围边界；
- 允许动作；
- 关键风险；
- 自审结论；
- 测试锚点；
- 完成汇报方式。

自审结论里的 Evidence 如果不满足质量门槛，视同未完成自审。

长任务、多步执行、阻塞恢复、Task Graph、Task Memory、Probe、完成闸门等细节，读取 `references/runtime.md`。

## 反例约束

不要这样做：

- 不要直接按用户原话实现；
- 不要只给一个解释，让用户自己发现偏差；
- 不要让非推荐项变成“更保守/更完整”的空泛陪衬；
- 不要问一串开放式问题；
- 不要问不会改变行为的问题；
- 不要为了流程完整输出低价值章节；
- 不要把实现后的总结伪装成 BRT；
- 不要在关键意图未对齐时开始写代码。

## 收口

每轮 BRT 简短收口：

- 已对齐的真实意图；
- 已排除的误解；
- 当前行为契约；
- 自审结论；
- 验证锚点；
- 剩余风险；
- 下一步。

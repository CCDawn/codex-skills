# BRT

## 核心目标

使用 BRT 的第一目标不是展示流程，而是主动对齐用户心里真正想要的东西。

BRT 要把“用户说出口的入口、感受或局部方案”低噪声地收敛成“可以实现、可以评审、可以测试、可以持续推进的行为契约和任务执行流”。

BRT 三段：

- 行为：锁定目标、使用者、触发条件、可观察结果、范围边界和失败路径。
- 评审：推荐合适视角，经用户确认后挑战行为设计。
- 测试：把场景映射到最小但足够的测试或验收证据。

BRT 同时提供 runtime 执行内核：

- 收敛：用多假设意图发现把模糊请求收束到可执行解释。
- 拆解：把长任务拆成有输入、输出、依赖和验证条件的 Task Graph。
- 执行：每一步执行前校验前置条件和歧义，执行中不扩展范围，执行后验证输出并更新状态。
- 记忆：维护任务意图、假设、决策、进度和未解决风险。

把 BRT 当成实现前闸门和执行中护栏，而不是实现后的包装总结。

## 心智对齐优先

所有 BRT 动作都服务于一个问题：当前理解是否贴近用户心里的真实目标。

优先级：

1. 对齐用户真实意图；
2. 主动揣测用户没说出口的目标、担心和取舍；
3. 给出可选建议，让用户通过选择完成对齐；
4. 找出会改变行为、范围、风险、测试或实现许可的缺口；
5. 形成可验证行为；
6. 再考虑 Task Graph、评审、测试和执行。

不要等待用户自己把需求说完整。用户表达模糊、含混、矛盾或带有宽泛词时，先主动揣测 2-3 个可能真实意图，再给推荐选择和追问。用户已经明确、风险很低、可逆且能本地查证时，用显式假设继续推进。

## 触发规则

当请求涉及以下任一内容时，主动使用 BRT：

- 预期行为、默认值、用户应该看到什么；
- 记忆、恢复、续做、再次打开；
- 状态流转、生命周期、重试、取消、回滚；
- 权限、隐私、密钥、安全、数据保留；
- 边界情况、失败路径、部分成功、冲突；
- 验收标准、BDD、Given/When/Then、怎么测试；
- agent 行为、工具选择、计划、交接、自主工作流；
- 长任务、多步执行、状态保持、任务恢复、执行计划、阻塞恢复；
- promotion/apply/rollback、持久化、公共 API 契约。

明显拼写修复、纯机械重构、依赖升级、没有行为歧义的小型内部 helper 改动，不要使用 BRT。

## 不可破坏的规则

- 先停在实现前；除非用户明确说跳过 BRT 或直接实现。
- 需求对齐不是固定步骤，而是贯穿式揣测、建议、追问和校准机制。
- 任一阶段出现影响行为、范围、风险、测试或实现许可的不确定性，都要暂停并追问。
- 允许一次问多个问题，但只能问高信号问题；默认 2-4 个，按影响排序，每个都给推荐答案和理由。
- 能从代码、测试、文档、配置、日志查到的，先查再问。
- 用户回答后，先更新当前 BRT 产物，再继续推进。
- 不要把用户原话直接当成需求；先翻译成可观察意图，再确认。
- 不要把 BRT 结果写成实现后的补票。
- 意图没有收敛前，不要直接进入实现。
- 长任务开始执行前，必须先拆成可验证的 Task Graph。
- 每个执行步骤前必须检查前置条件和剩余歧义；执行后必须验证输出并更新任务状态。
- 不要在执行中静默扩展范围；缺失约束只能显式假设或追问，不能暗中补全。

## 中文基线

默认使用中文结构和中文解释。代码、配置键、CLI 命令、环境变量、文件路径、API 名称、产品标识符保留原文。

统一使用这些中文标签：行为、评审、测试、需求对齐、追问、范围边界、推荐评审视角、决策记录、测试锚点、任务契约、任务状态、Task Graph、Task Memory、执行闸门、完成闸门、BRT 结果、下一步。

严重级别统一为：阻塞、重要、可选。

## 意图发现

用户通常说的是入口、感受或局部方案，不一定是真实需求。先把用户原话翻译成可验证意图。

必须区分：

- 用户原话：用户实际说了什么；
- agent 推断意图：用户可能真正想解决什么问题，特别是没明说但明显影响目标的动机；
- 心智画像：用户可能在担心什么、嫌什么麻烦、想避免什么失败、希望最终看到什么；
- 多假设路径：默认显式给出 2-3 个候选意图，让用户选择或修正；
- 每个候选假设至少评估 Goal、Expected Output、Required Capability；输出时保持短句；
- 关键词风险：哪些词可能过宽、过窄、用错或带有隐藏假设；
- 推荐解释：建议按哪个意图推进；
- 待确认点：当前最影响行为边界的一组高信号问题。

遇到“所有”“智能”“自动”“优化”“更好用”“闭环”“记忆”“统一”“简单一点”等宽泛词，不要照字面实现。先翻译成可观察行为，并向用户确认。

意图发现的追问要降低用户成本：给出你的推荐解释，让用户确认或纠正，而不是问“你具体想要什么”。

揣测不是擅自决定。揣测必须带着可纠正接口：

- 先说“我理解你可能真正想解决的是 X/Y/Z”；
- 再说“我建议选 X”；
- 然后给 2-4 个高信号问题或选项；
- 最后允许用户直接回答选项、改写目标或说“按你建议来”。

## 信号与噪声过滤

BRT 要主动减少噪声冗余信息。不要把所有可能相关的信息都问出来、列出来或塞进检查点。

高信号信息包括：

- 会改变用户最终可观察结果的信息；
- 会改变范围边界的信息；
- 会改变读写、删除、发布、回滚、权限或安全风险的信息；
- 会改变测试策略或验收证据的信息；
- 会改变是否允许继续实现的信息。

低信号噪声包括：

- 不会改变本轮行为的背景闲聊；
- 可以从代码、文件、日志、配置或上下文查到的信息；
- 只是实现偏好但不影响行为契约的信息；
- 多个等价命名、措辞或展示顺序；
- 过早的完整方案、长清单或模板填空。

处理规则：

- 能查就先查，不要问；
- 能用低风险假设继续，就声明假设并继续；
- 只有问题答案会改变行为、范围、风险、测试或许可时才追问；
- 每轮默认问 2-4 个高信号问题；低风险小任务可只问 0-1 个，高风险任务可问到 5 个；
- 问题尽量做成选项，让用户选择，而不是要求用户自己组织需求；
- 每个问题必须给推荐答案；
- 输出默认短，只有当用户要求、风险高或即将实现时才展开完整 BRT。

## 追问决策

追问的目的不是收集更多信息，而是校准用户心里真正想要的结果。BRT 应主动替用户提出候选答案，让用户选择、纠正或确认。

必须追问：

- 推荐理解和备选误解会导致不同的用户可观察结果；
- 答案会改变是否读、写、删除、发布、回滚或触碰权限/安全边界；
- 答案会改变数据保留、状态迁移、默认值或失败路径；
- 答案会改变测试层级或验收证据；
- 不问就可能做出用户心里明确不想要的结果。

不要追问：

- 答案只影响实现细节，不影响行为契约；
- 可以从本地上下文查到；
- 可以用低风险可逆探针确认；
- 可以声明低风险假设并继续；
- 用户已经表达出“先做一个能用版本”的偏好。

多问题追问规则：

- 同一轮问题按影响排序：目标 > 范围 > 风险/权限 > 输出形态 > 测试验收；
- 每个问题只解决一个决策点；
- 每个问题给 2-3 个互斥选项，并标出推荐选项；
- 允许最后加一句“你也可以直接说按推荐来”；
- 不要问超过用户当前决策能力的问题；如果问题太多，先问前 3 个。

停止追问的条件：

- 能用一句话说清“用户真正想要的是 X，不是 Y”；
- 能写出最小主路径和一个关键边界；
- 能说明本轮不做什么；
- 能给出验证锚点；
- 剩余不确定性都可以低风险假设或后续修正。

追问要像校准面板：只放会改变结果的旋钮。可以一次放多个旋钮，但每个都必须高信号、可选择、可行动。

当满足以下条件，就停止深挖并继续 BRT：

- 能写出一个主路径 Given/When/Then；
- 能写出一个失败或边界路径；
- 能明确本轮不做什么；
- 能确定验证证据；
- 至少一个假设路径稳定，且剩余假设风险可以显式记录；
- 剩余问题都是低风险实现细节。

## 任务状态

只在状态会影响下一步动作时输出任务状态。可用状态：INTENT_DISCOVERY、INTENT_CONVERGENCE、PLANNING、EXECUTING、VERIFYING、BLOCKED、COMPLETED。

长任务、多步执行、阻塞恢复或 agent 执行行为需要完整状态机时，读取 `references/runtime.md`。

## 收敛判定

意图发现后必须给出收敛判定：

- CONVERGED：意图和范围足够稳定，可以进入规划。
- NEED_ASSUMPTION：可继续，但必须声明 assumed intent、assumed constraints 和 risk of assumption。
- NEED_PROBE：所有假设都不稳定，必须先做低风险探针或追问。

下一步动作只能选择一个：

- ENTER_PLANNING：进入任务拆解。
- ASK_CLARIFYING_QUESTIONS：提出 2-4 个会改变行为、范围、风险、测试或实现许可的高信号问题。
- RUN_PROBE：执行低风险、可观察、可回滚的探针。

## 意图校准失败处理

如果用户说“不对”“不是这个意思”，或表示“我也说不清”，不要要求用户重写完整需求。

先承认推断失效，再定位偏差：

- 重点错：agent 把核心问题放错了；
- 范围错：agent 理解得过宽或过窄；
- 对象错：agent 选错了用户、数据、页面、模块或流程；
- 结果错：agent 误判了用户想观察到的结果；
- 操作方式错：agent 把只读理解成编辑、把辅助理解成自动执行等。

恢复流程：

1. 复述被否定的推断；
2. 给出 2-3 个候选意图，让用户选“更接近”，而不是要求用户精确定义；
3. 如果用户仍不确定，提出一个最小可逆探针；
4. 探针必须低风险、可观察、可回滚，不做破坏性写入、删除、自动发布或长期承诺；
5. 用户选择候选意图或接受探针后，回写需求账本，再继续 BRT。

探针不是正式实现。它用于验证方向，必须写清：做什么、不做什么、如何观察是否接近真实意图。

## 第一响应规则

如果用户表达模糊、使用宽泛词、提出局部方案，或看起来不清楚完整需求，第一条回复必须先做意图发现和一个确认问题。不要先给完整方案。

第一响应至少包含：

- 用户原话；
- 推荐理解：你判断用户心里真正想达成什么；
- 候选意图：2-3 个可能真实意图，并标出推荐项；
- 备选误解风险：最可能误解的相邻意图；
- 关键信号：为什么这样理解，来自用户措辞、上下文、代码或任务形态；
- 状态判定：CONVERGED、NEED_ASSUMPTION 或 NEED_PROBE；
- 下一步动作：ENTER_PLANNING、ASK_CLARIFYING_QUESTIONS 或 RUN_PROBE。

显式展开 2-3 个候选意图，让用户能选择或纠正。每个候选只写 Goal、Expected Output、Required Capability。

如果需要追问，提出 2-4 个高信号问题；每个问题给 2-3 个选项和推荐答案。不要附带低价值顺手问题。

如果需求已经足够清楚，也要简短写出采用的意图假设，再进入 Quick BRT 或 Full BRT。

## 贯穿式追问

贯穿式追问是 BRT 的中断机制。它不只发生在开头。

触发条件：

- 行为不清楚：目标、使用者、触发、结果、范围边界无法写清；
- 方案不清楚：推荐方案依赖未确认的产品、架构、数据或安全假设；
- 场景不清楚：无法写出明确 Given/When/Then，或 Then 不可观察；
- 评审不清楚：某个视角发现会改变设计的风险或缺口；
- 测试不清楚：不知道测试层、成功证据或验收方式；
- 检查点不清楚：允许动作、验证方式、限制或完成汇报不明确；
- 实现中不清楚：代码现实与原设计冲突，或出现新的行为分叉；
- 收口不清楚：剩余风险、开放问题、下一步或是否完成不明确。

追问格式：

```text
需求对齐:
- 当前理解: [现在已经确认的事实]
- 未决分歧: [当前不确定的点]
- 为什么重要: [会影响行为/范围/风险/测试/实现许可的哪一块；如果不影响，就不要问]
- 会更新: [需求账本/示例/评审/测试映射/任务契约/实施目标/收口]

追问:
- Q1: [最高影响问题]
  - 选项: A / B / C
  - 推荐答案: [我的建议]
  - 原因: [简短原因]
- Q2: [第二影响问题]
  - 选项: A / B / C
  - 推荐答案: [我的建议]
  - 原因: [简短原因]
```

更短的默认追问格式：

```text
需求对齐:
- 我猜你真正想要的是: [推荐理解]
- 你也可能是在要: [候选 B / 候选 C]
- 我的建议: [推荐选择]

追问:
- 请选这几个关键项，或直接说“按推荐来”:
  1. [问题一]: A / B(推荐) / C
  2. [问题二]: A(推荐) / B / C
  3. [问题三]: A / B / C(推荐)
```

如果剩余歧义低风险，写明假设、影响范围和后续可修正点，然后继续。

## 输出预算

默认紧凑输出，不要把流程做成表演。

- INTENT_DISCOVERY / INTENT_CONVERGENCE：默认展示 2-3 个候选意图、推荐理解、关键信号、2-4 个高信号选择题、状态判定和下一步动作。
- PLANNING：默认展示范围边界、最小 Task Graph 和测试锚点；中高风险再展开需求账本、评审视角、测试映射和检查点。
- EXECUTING：最小输出当前步骤、执行闸门、结果和状态更新。
- VERIFYING：结论式输出 expected vs actual、验证证据、副作用检查和下一状态。
- 小型低风险任务：输出推荐理解、显式假设、必要选择题、行为摘要、测试锚点和下一步动作。
- 中等任务：输出需求账本摘要、Task Graph、评审结论、测试映射和检查点。
- 高风险任务：完整输出示例、场景、Task Memory、决策记录、任务契约和收口。

需要完整模板、反例或示例时，读取 `references/templates.md`。需要长任务 runtime 细则时，读取 `references/runtime.md`。需要正式 Gherkin 语法时，读取 `references/gherkin.md`。

## 模式选择

Quick BRT 和 Full BRT 的选择规则：

- 用户明确确认快速模式：使用 Quick BRT。
- 用户说“继续/开始/做吧”但没选模式：低风险任务默认 Quick BRT，并说明假设。
- 涉及安全、数据丢失、权限、迁移、公共 API、架构、持久化、回滚或长期契约：默认 Full BRT。
- Quick BRT 只减少篇幅，不减少贯穿式追问和实现前检查点。

## 主流程

1. 读上下文：附近代码、测试、文档、最近提交、`AGENTS.md`、`CONTEXT.md`、issues、计划或已有 feature 文件。
2. 设置任务状态为 INTENT_DISCOVERY。
3. 做高信号意图发现：区分用户原话、2-3 个候选意图、推荐理解、备选误解风险、关键信号和关键追问。
4. 显式给出可选择的候选意图，除非用户目标已经完全清楚。
5. 过滤噪声：删掉不会改变行为、范围、风险、测试或实现许可的问题和解释。
6. 做收敛判定：CONVERGED、NEED_ASSUMPTION 或 NEED_PROBE。
7. 如果用户否定推断或仍不确定，进入意图校准失败处理；如果全部假设 unstable，先 RUN_PROBE。
8. 收敛后设置任务状态为 PLANNING。
9. 建立或更新需求账本；默认只输出摘要，完整账本只在中高风险任务输出。
10. 写范围边界：本轮覆盖什么，不覆盖什么。
11. 比较 2-3 个方案；输出推荐方案，并给用户可选的相邻方案。
12. 先写具体示例，再写一般规则；跨 UI、CLI、工作流、持久化或 agent 状态时，用 Given/When/Then。
13. 对多步或长任务拆 Task Graph；单步低风险任务只写下一步动作和验证锚点。
14. 按风险推荐评审视角；低风险任务不需要把 3-5 个视角全部展示。
15. 映射测试：选择最低但足够的测试层，覆盖主路径和一个关键边界。
16. 即将改代码时展示 BRT 检查点、执行闸门和任务契约。
17. 用户确认实现后，设置任务状态为 EXECUTING，按执行循环推进：step gate、execute step、verify output、update Task Memory。
18. 需要验证时设置任务状态为 VERIFYING；验证失败或缺少必要输入时设置为 BLOCKED。
19. 所有 critical steps 完成且验证通过后设置任务状态为 COMPLETED。
20. 收口时说明已交付行为、验证结果、开放问题、剩余风险和下一步建议。

## 需求账本

对非单点小改，必须维护需求账本。它不是只在开头填写一次；评审、测试、实现或收口中发现新事实时，要回写。

需求账本包含：

- 用户原话：用户实际表达；
- agent 推断意图：用户可能真正想解决的问题；
- 已确认意图：用户确认后的意图；
- 用户目标：最终要解决什么问题；
- 使用者/触发者：谁使用，或什么事件触发；
- 可观察结果：用户、日志、文件、接口、测试或系统状态能看到什么；
- 数据/状态边界：读、写、展示、持久化哪些内容，排除哪些内容；
- 操作边界：只读、编辑、删除、回滚、部署、提交等是否允许；
- 失败路径：缺数据、旧格式、权限不足、运行失败、部分成功时怎么表现；
- 验证证据：用测试、日志、浏览器检查、人工验收还是真实设备验证。

账本有关键空项时，先查本地上下文；查不到就追问最关键的一项。

## Task Graph

只在任务确实多步、跨文件、长期执行或有明显风险时拆 Task Graph。每步至少写 output 和 verification condition；低风险单步任务只写下一步动作和验证锚点。

完整 Task Graph 规则见 `references/runtime.md`。

## Task Memory

连续任务才维护 Task Memory，默认只记录：已收敛意图、显式假设、关键决定、步骤进度、未解决风险。不要为一次性低风险小任务输出 Task Memory。

完整 Task Memory 规则见 `references/runtime.md`。

## 评审

推荐视角要从需求账本里的风险推导出来，不要固定套模板。

默认基线视角：

- 核心用户；
- 维护者；
- 测试/QA。

按风险补充：新用户、高级用户、产品负责人、领域专家、安全/隐私、运维/支持、后续集成方/插件作者。

得到用户确认后，每个视角至少说明：

- 关注点；
- 发现；
- 为什么重要；
- 推导结果。

只要比较过备选方案、出现阻塞发现、评审改变设计、排除诱人的相邻功能，或做了非显然的架构/数据/安全/持久化/API 选择，就写决策记录。

## 测试

选择能证明行为的最低测试层：

- 单元测试：隔离规则、解析、格式化、策略、状态决策；
- 集成测试：持久化、CLI 提示流、文件系统、多模块协作、agent 工作流；
- 端到端测试：真实浏览器/终端流程，或只有完整产品链路才能看到的行为；
- 人工验收：视觉、探索性或暂时不适合自动化的行为。

测试映射必须选定具体策略，不要写成“或者这样测、或者那样测”。测试数量保持最小，但要覆盖主路径和至少一个重要失败/边界路径。

## 实现闸门

改代码前必须展示 BRT 检查点，除非用户明确要求跳过。

检查点前先做最后一次不确定性扫描；如果任务契约、验证方式、范围边界、测试锚点、Task Graph、允许动作或当前任务状态仍不清楚，先追问。

检查点必须包含：

- 任务状态；
- 锁定行为；
- 范围边界；
- Task Graph；
- 评审视角和阻塞问题；
- 测试锚点；
- 实施目标；
- 执行闸门；
- 任务契约。

执行闸门包含：

- prerequisites：执行前必须存在的文件、上下文、权限、测试或用户确认；
- no ambiguity check：当前步骤是否还有会改变行为或风险的歧义；
- step scope：本次只执行哪个 Task Graph 步骤；
- expected output：本步骤的输出契约；
- verification command or evidence：如何验证本步骤。

任务契约包含：

- 目标：用户最终想达成什么；
- 上下文：相关代码、日志、文档或历史决策；
- 限制：哪些文件、行为、接口不能随便改；
- 允许动作：只分析、写计划、直接实现、实现并部署、还是提交变更；
- 验证方式：测试命令、手工验收、日志证据、浏览器检查或设备验证；
- 完成汇报：根因、改动文件、验证结果、剩余风险。

## 执行循环

执行时只做当前步骤，执行前确认无关键歧义，执行后验证 output contract 并更新状态。不要在执行中扩展范围。

长任务执行循环见 `references/runtime.md`。

## Probe 机制

当意图、环境或实现路径不稳定时，优先使用低风险 probe，而不是直接实现。Probe 必须可观察、可回滚、无破坏性；Probe 不是正式实现。

完整 Probe 结构见 `references/runtime.md`。

## 完成闸门

任务只有在关键行为已交付、验证通过、没有未解决阻塞、且用户意图已满足或显式更新时，才能进入 COMPLETED。

完整完成闸门见 `references/runtime.md`。

## 反例约束

不要这样做：

- 不要一次抛出多个顶部问题；
- 不要在用户说“不对”后继续逼用户完整重写需求；
- 不要先给方案，再倒问用户需求；
- 不要按“所有、智能、自动、优化”等宽泛词照字面实现；
- 不要把用户提出的局部方案误当成最终目标；
- 不要只给单一意图解释，而不比较保守路径和标准路径；
- 不要在全部假设 unstable 时进入规划或实现；
- 不要没有 Task Graph 就开始长任务实现；
- 不要跳过执行前 prerequisites 和 no ambiguity check；
- 不要把 optional path 偷偷混进当前步骤；
- 不要执行后不更新 Task Memory；
- 不要把最小探针伪装成正式实现；
- 不要用“后续跟进”替代具体下一步；
- 不要把 Quick BRT 当成跳过评审和测试映射；
- 不要把不确定项塞进检查点；
- 不要在实现中发现新行为分叉后继续硬写代码；
- 不要只在实现后补一段“BRT 结果”。

## 收口

每次 BRT 都要简短收口。连续开发、调试、优化或 skill 设计任务，必须把本轮结果接回主线目标。

收口至少说明：

- 最终任务状态；
- 已锁定或已交付的行为；
- 评审阻塞项是否解决；
- 实际验证或测试锚点；
- Task Graph 中 critical path 的完成情况；
- Task Memory 中仍保留的假设、决策和风险；
- 开放问题和剩余风险；
- 具体、可执行的下一步建议。

如果没有合理下一步，明确写“无明确下一步，当前任务可收口”。

## 完成检查

结束前确认：

- 变更后会出现什么行为；
- 用户或系统如何观察到它；
- 范围边界排除了什么；
- 哪些示例证明设计；
- 哪些视角挑战过设计；
- 哪些决定被记录，为什么；
- 哪些测试保护行为；
- 当前任务状态是什么，是否符合下一步动作；
- Task Graph 是否覆盖 critical path，并为每步定义验证条件；
- Task Memory 是否记录 intent、assumptions、decisions、step progress、unresolved risks；
- 每个执行步骤是否完成 prerequisites、no ambiguity check、expected vs actual 和副作用检查；
- 是否满足完成闸门，或明确处于 BLOCKED 并提出必要的高信号选择题；
- 被否定或不确定的意图是否已校准，或已转成最小可逆探针；
- 全流程出现的新不确定性是否已追问、查证或显式假设；
- 连续任务是否接回主线目标并给出下一步。
